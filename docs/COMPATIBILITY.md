# Git compatibility

`version` targets practical file-level interoperability with ordinary Git repositories. It stores Git-compatible data where documented, but it does not promise Git command-line parity.

| Feature | Status | Notes |
|---|---:|---|
| Loose objects | supported | Read/write commit, tree, blob objects. |
| Pack reading | supported | Pack/index and delta handling in supported flows. |
| Pack writing | supported | Used by repack and transport paths. |
| Local clone/fetch/push | supported | Local paths and `file://`; `file://localhost/...` is local, other file URL authorities are rejected, percent escapes are decoded for access, and stored URLs are preserved as configured. Local `push --tags` is supported. |
| Smart HTTP | supported/limited | Smart protocol scope; HTTPS remotes negotiate HTTP/2 through `HttpClient` ALPN (multiplexed streams, verified end-to-end against a live HTTP/2 git server) with HTTP/1.1 fallback, while h2c/plain-HTTP HTTP/2 remains unsupported (see ROADMAP). Branch push and `push --tags` are supported (tags refuse to overwrite a differing remote tag). Normal tests do not use public internet. |
| SSH | supported/limited | Native SSH via the `ssh_lib` library (no external `ssh` binary); branch push and `push --tags` are supported. Connection settings (HostName aliases, Port, User, IdentityFile, known_hosts) are resolved from `~/.ssh/config`; host-key verification, identity files, and agent use are handled by `ssh_lib`. |
| Reflogs | supported/limited | Core operations and stash stack; not full Git nuance. |
| Submodules | supported/limited | Init/update/status/foreach/sync/deinit for conservative `.gitmodules` and gitlinks. `foreach` runs a shell command in each populated submodule with git's `$name`/`$sm_path`/`$displaypath`/`$sha1`/`$toplevel` environment; `sync` copies `.gitmodules` URLs into `submodule.<name>.url` and the submodule's `remote.origin.url`; `deinit` empties the working tree and removes the submodule config (keeping `.gitmodules`) — verified against git. Merge can fast-forward checked-out submodule worktrees to merged gitlink commits, fetching missing target commits and updating nested submodules when recursion is enabled, while refusing dirty tracked/index/untracked submodule state before updating. |
| Worktrees | supported/limited | Linked worktrees with shared objects/refs and isolated per-worktree state. |
| Branch/tag/status/diff/log/show | supported/limited | Practical deterministic behavior, including top-level `merge` with common fast-forward/message/no-commit/squash options, upstream default targets, merge message file/log/signoff/cleanup/edit handling, selected merge config defaults including `branch.<name>.mergeOptions`, concrete `resolve`/`subtree`/`octopus` strategy semantics, `MERGE_AUTOSTASH` recovery, multi-target squash state, line-level text auto-merge with one-sided regular-file mode preservation and POSIX worktree executable-bit materialization plus clean merge symlink materialization that attempts host symlink creation when enabled and uses the `core.symlinks=false` plain-file fallback, identical-content/mode-only auto-resolution, default and selected diff-algorithm multi-hunk matching including explicit Myers, deterministically ordered recursive virtual bases for criss-cross merges with materialized text, binary, delete/modify, directory/file, rename/delete, rename/rename, and directory-rename virtual-base conflicts, subtree prefix rewriting, similarity rename detection with `merge.renames` and `merge.renameLimit` config defaults and explicit option overrides, opt-in copy-base detection for add/add paths including `find-copies-harder` aliases, configurable `merge.directoryRenames` handling including ambiguous split-directory conflicts, same-path rename/rename including regular-file mode-only changes and rename/add collision staging, scoped built-in and configured external merge drivers from worktree and info attribute files with `!merge` resets and shell-quoted path/label placeholders, Git-like driver environment, fatal-status and missing-result propagation with internal temp cleanup, and built-in and delegated recursive-driver handling for virtual bases, `merge.renormalize` config defaults with explicit option overrides, Git-style unmerged index stages including clean multi-head no-commit state, basic side-order-independent rerere preimage/postimage reuse, `rerere.autoupdate` enablement, exact-preimage lookup for merge, and rerere preimage/postimage recording plus recorded-resolution reuse for rebase/cherry-pick/revert replay conflicts, byte-exact git success output (the `Updating <old>..<new>` fast-forward line, `Fast-forward`/`Merge made by the 'ort' strategy.`, the `--stat` diffstat plus `--summary` create/delete-mode block honoring `merge.stat`, and octopus `Fast-forwarding to:`/`Trying simple merge with` progress), Git-style typed conflict summary diagnostics for recorded merge conflicts, including rename/delete, rename/rename, and directory-rename file-location conflicts inferred from merge-base state, without an extra internal conflict-summary error, and Git merge-state continuation/abort over the branch-integration engine; target signature verification delegates to local Git/GPG tooling, signed merge commit creation delegates to local `gpg`; merge success output byte-matches git, and conflict output byte-matches git across the conflict types (content, add/add, modify/delete, delete/modify, rename/delete, rename/rename, rename/add, file/directory with `<path>~<label>` rename, and unrelated histories), verified differentially. |
| Ignore files | supported | Repository `.gitignore`, local `.git/info/exclude`, `core.excludesFile` from Git's system/global/local config stack, and Git's default global ignore file are honored for untracked-file filtering, `check-ignore`, `status --ignored[=traditional|matching|no]`, and `stage --force`, including quoted path values, `%(prefix)` interpolation, Git config environment overrides including `GIT_CONFIG_COUNT`, `[include] path` config files, matching `[includeIf "gitdir:..."] path` config files, matching `[includeIf "onbranch:..."] path` config files, and matching `[includeIf "hasconfig:remote.*.url:..."] path` config files; source precedence is `core.excludesFile`, then `info/exclude`, then `.gitignore`. |
| Rebase/cherry-pick/revert | supported/limited | Rebase remains linear/non-merge with continue/abort; cherry-pick and revert also support root commits and merge commits with `--mainline`. |
| Stash | supported/limited | Tracked work plus optional untracked/ignored files, pathspec-limited push/create/apply/pop, store with optional message, pathspec-filtered read-only show/patch inspection, branch-from-stash, and clear. |
| Sparse checkout | supported | `sparse-checkout set`/`add`/`list`/`status`/`reapply`/`init`/`disable` (and the `sparse` alias) match git. `set`/`add` default to cone mode: they write git's cone patterns (`/*`, `!/*/`, `/dir/`, and `!/dir/*/` ancestor entries), set `core.sparseCheckout`/`core.sparseCheckoutCone`, materialize the working tree, and set git skip-worktree bits on excluded index entries (emitting a version-3 index) — so `git status`/`ls-files -t` round-trip a version-created sparse checkout, verified against git. `--no-cone` writes raw patterns; `disable` keeps the pattern file. On the read side git's cone-mode repositories (including a version-3 sparse index whose `sdir` placeholders are expanded) are operated on directly, and sparse-excluded paths are not reported as deleted by `status`/`diff`. Mutating commands still write a full (non-sparse-directory) index rather than preserving the `sdir` sparse-index shape. |
| Shallow clone/fetch | supported | Depth-based shallow clone/fetch over smart (HTTP/SSH) transports, plus `fetch --deepen N` (relative deepening via the `deepen-relative` capability) and `fetch --unshallow` (fetch complete history, remove `.git/shallow`); the client echoes its current shallow boundary and applies the server's `shallow`/`unshallow` responses. Verified against a live `git http-backend` server (boundary and history match git, `git fsck` clean). Partial-clone behavior is covered separately. |
| Hooks | supported/limited | `pre-commit`, `commit-msg`, `pre-merge-commit`, `post-commit`, `post-merge`, `post-checkout`, `pre-push`. |
| LFS | supported/limited | Merge worktree writes smudge valid pointer blobs to local `.git/lfs/objects` media when present, and can fetch missing media from local/file LFS stores, HTTP(S) Git LFS batch download endpoints, or SSH `git-lfs-authenticate` followed by the advertised HTTP(S) batch endpoint before materializing the merge result. Staging paths marked `filter=lfs` in root `.gitattributes` or `.git/info/attributes` stores media in `.git/lfs/objects` and writes canonical pointer blobs; restore/checkout materializes pointer blobs to media for `filter=lfs` paths when media is cached or fetchable through the same LFS transfer paths. Paths without `filter=lfs` still treat LFS pointers as ordinary files/blobs. |
| Partial clone | supported/limited | Repository format version 1 with `extensions.partialClone` is accepted. `clone --filter=SPEC` creates a partial clone: over HTTP/SSH the arbitrary `SPEC` is sent verbatim when the server advertises the upload-pack `filter` capability; over local transport version evaluates `blob:none` and `blob:limit=<n>` directly (selective object copy, richer specs are rejected for local sources). The clone is configured as a partial clone (`extensions.partialClone`, promisor remote, `partialclonefilter`). Missing objects are then lazily fetched once from the promisor by object id over local (single-object copy), HTTP, or SSH transports without updating refs. Promisor pack sidecars without a configured partial-clone remote still report the promisor limitation. |
| worktreeConfig | supported | `extensions.worktreeConfig` is accepted; the per-worktree `config.worktree` is layered over the common `config` when reading configuration (per-worktree values override the common config, and absent keys fall back to it). Writing to `config.worktree` (`config --worktree`) is not yet provided. |
| Config scope & includes | supported | Reading the effective config (`config get`/`list`/`keys`) reads git's full scope stack in order — system (`/etc/gitconfig` or `GIT_CONFIG_SYSTEM`, suppressed by `GIT_CONFIG_NOSYSTEM`), global (`$XDG_CONFIG_HOME/git/config` then `~/.gitconfig`, or `GIT_CONFIG_GLOBAL` replacing both), local `.git/config`, and per-worktree `config.worktree` — and follows `[include]` and matching `[includeIf]` directives (`gitdir:`, `gitdir/i:`, `onbranch:`, `hasconfig:remote.*.url:`), resolving the include `path` relative to the including file (with `~` expansion) and keeping the directive as a readable key; a single-valued `get` resolves to the last matching value, as git does. `GIT_CONFIG_COUNT`/`GIT_CONFIG_KEY_<n>`/`GIT_CONFIG_VALUE_<n>` env injection is not yet consulted; writes still target the local config only. |

Supported workflows include ordinary init/stage/save/status/log, Git inspection via `git fsck --strict`, local/file remotes, configured HTTP(S)/SSH remotes, practical branch/tag/replay/stash workflows, sparse checkouts, linked worktrees, and conservative submodule updates.

URL rewriting via `url.<base>.insteadOf` / `pushInsteadOf` is applied when resolving fetch and push remote URLs (longest matching prefix wins; `pushInsteadOf` takes precedence for push), and submodule update recurses into nested submodules. Partially supported workflows include reflog parity, worktree administrative parity beyond the documented per-worktree state, advanced external merge driver edge cases, exact Git ort output/diagnostic parity beyond the documented success and conflict-summary diagnostics, `GIT_CONFIG_COUNT` environment config injection for the effective `config` reader (the file-based system/global/local/worktree scopes and their `[include]`/`[includeIf]` directives are read), and advanced pathspec magic beyond the documented literal/glob/top/icase/exclude/attr/set/unset/unspecified/value forms.

Windows portability is gated in the platform abstraction: backslash separators normalize to `/`, drive (`C:\`) and UNC paths are recognized, reserved device names / invalid filename characters / trailing dots and spaces are rejected, path collisions are case-insensitive, `core.filemode` defaults to `false`, symlink checkout falls back to plain link-target files (real symlink creation needs host privileges), and a fully-qualified path of at least MAX_PATH (260) characters is given the `\\?\` extended-length prefix (`\\?\UNC\` for UNC). Native-Windows credential-manager and PowerShell/`.cmd` hook execution beyond the POSIX-style shell model (as provided by a Git-for-Windows/MSYS environment) are not separately implemented.

Interactive rebase (`rebase -i`) supports the full action set — pick/drop/reorder/squash/fixup/reword/edit/exec — via the sequence editor. Root rebase is supported via `rebase --root --onto NEWBASE` (bare `--root` is not). Merge-preserving rebase is supported via `rebase --rebase-merges` for both two-parent and octopus (>= 3 parent) merges (recreated by git's iterated merge-octopus strategy); a conflict inside an octopus aborts rather than pausing, and `--preserve-merges` is not supported. Unsupported workflows include server-side hook management and a website.


## Archive export

Supported/limited: `version archive` can export committed trees as TAR or ZIP, preserving file bytes, CRLF data, binary payloads, directory structure, symlink metadata, and executable metadata where the archive format supports it. Gitlink/submodule entries are represented as placeholder files and are not recursively archived; Git symlink entries are represented as symlinks in TAR/ZIP metadata. Byte-for-byte parity with `git archive` is not required. `tar.gz`/`tgz` are produced with the integrated Ada Zlib gzip path, and any other format (e.g. `tar.xz`, `tar.bz2`) is produced through git's tar-filter mechanism (`tar.<format>.command`); a format with no built-in support and no configured filter is rejected as git does. Supported archive prefixes are safe relative root paths applied to emitted entry names after pathspec matching. Unsupported: unsafe archive prefixes and remote archive streaming.

## Unsupported diagnostics

SHA-256 object-format repositories (`extensions.objectFormat = sha256`) are supported: `init --object-format=sha256` creates them, and they are read and written (local and smart HTTP/SSH transport, format negotiated via the `object-format=sha256` capability), byte-compatible with system git. A repository is a single object format for its lifetime.

Reftable ref storage (`extensions.refStorage = reftable`, `repositoryformatversion = 1`) is supported for reading and writing: version reads git-created reftable stacks, writes git-readable tables, stores the reflog in reftable log blocks, and `init --ref-format=reftable` creates such repositories (verified `fsck`-clean and `git reflog`-readable against git 2.54). Writes append one small table per transaction with geometric auto-compaction keeping the stack bounded, preserving tombstones and reflog history through merges.

Unsupported features must fail with precise, stable diagnostics. The cases include promisor sidecars without a configured partial-clone remote, unavailable HTTP/3 support, h2c upgrade, server push, unsupported remote URL schemes, and SSH upload-pack/receive-pack streaming limitations.
