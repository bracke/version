# Git compatibility

`version` targets practical file-level interoperability with ordinary Git repositories. It stores Git-compatible data where documented, but it does not promise Git command-line parity.

| Feature | Status | Notes |
|---|---:|---|
| Loose objects | supported | Read/write commit, tree, blob objects. |
| Pack reading | supported | Pack/index and delta handling in supported flows. |
| Pack writing | supported | Used by repack and transport paths. |
| Local clone/fetch/push | supported | Local paths and `file://`; `file://localhost/...` is local, other file URL authorities are rejected, percent escapes are decoded for access, and stored URLs are preserved as configured. Local `push --tags` is supported. |
| Smart HTTP | supported/limited | Smart protocol scope; HTTPS remotes prefer HTTP/2 through `HttpClient` ALPN with HTTP/1.1 fallback, while h2c/plain HTTP/2 remains unsupported. Branch push and `push --tags` are supported (tags refuse to overwrite a differing remote tag). Normal tests do not use public internet. |
| SSH | supported/limited | Native SSH via the `ssh_lib` library (no external `ssh` binary); branch push and `push --tags` are supported. Connection settings (HostName aliases, Port, User, IdentityFile, known_hosts) are resolved from `~/.ssh/config`; host-key verification, identity files, and agent use are handled by `ssh_lib`. |
| Reflogs | supported/limited | Core operations and stash stack; not full Git nuance. |
| Submodules | supported/limited | Init/update/status for conservative `.gitmodules` and gitlinks; merge can fast-forward checked-out submodule worktrees to merged gitlink commits, fetching missing target commits and updating nested submodules when recursion is enabled, while refusing dirty tracked/index/untracked submodule state before updating. |
| Worktrees | supported/limited | Linked worktrees with shared objects/refs and isolated per-worktree state. |
| Branch/tag/status/diff/log/show | supported/limited | Practical deterministic behavior, including top-level `merge` with common fast-forward/message/no-commit/squash options, upstream default targets, merge message file/log/signoff/cleanup/edit handling, selected merge config defaults including `branch.<name>.mergeOptions`, concrete `resolve`/`subtree`/`octopus` strategy semantics, `MERGE_AUTOSTASH` recovery, multi-target squash state, line-level text auto-merge with one-sided regular-file mode preservation and POSIX worktree executable-bit materialization plus clean merge symlink materialization that attempts host symlink creation when enabled and uses the `core.symlinks=false` plain-file fallback, identical-content/mode-only auto-resolution, default and selected diff-algorithm multi-hunk matching including explicit Myers, deterministically ordered recursive virtual bases for criss-cross merges with materialized text, binary, delete/modify, directory/file, rename/delete, rename/rename, and directory-rename virtual-base conflicts, subtree prefix rewriting, similarity rename detection with `merge.renames` and `merge.renameLimit` config defaults and explicit option overrides, opt-in copy-base detection for add/add paths including `find-copies-harder` aliases, configurable `merge.directoryRenames` handling including ambiguous split-directory conflicts, same-path rename/rename including regular-file mode-only changes and rename/add collision staging, scoped built-in and configured external merge drivers from worktree and info attribute recursive-driver handling for virtual bases, `merge.renormalize` config defaults with explicit option overrides, Git-style unmerged index stages including clean multi-head no-commit state, basic side-order-independent rerere preimage/postimage reuse, `rerere.autoupdate` enablement, exact-preimage lookup for merge, and rerere preimage/postimage recording plus recorded-resolution reuse for rebase/cherry-pick/revert replay conflicts, Git-style already-up-to-date/fast-forward/clean-merge/no-commit success text, Git-style typed conflict summary diagnostics for recorded merge conflicts, including rename/delete, rename/rename, and directory-rename file-location conflicts inferred from merge-base state, without an extra internal conflict-summary error, and Git merge-state continuation/abort over the branch-integration engine; target signature verification delegates to local Git/GPG tooling, signed merge commit creation delegates to local `gpg`, and output is not exact Git output. |
| Ignore files | supported | Repository `.gitignore`, local `.git/info/exclude`, `core.excludesFile` from Git's system/global/local config stack, and Git's default global ignore file are honored for untracked-file filtering, `check-ignore`, `status --ignored[=traditional|matching|no]`, and `stage --force`, including quoted path values, `%(prefix)` interpolation, Git config environment overrides including `GIT_CONFIG_COUNT`, `[include] path` config files, matching `[includeIf "gitdir:..."] path` config files, matching `[includeIf "onbranch:..."] path` config files, and matching `[includeIf "hasconfig:remote.*.url:..."] path` config files; source precedence is `core.excludesFile`, then `info/exclude`, then `.gitignore`. |
| Rebase/cherry-pick/revert | supported/limited | Rebase remains linear/non-merge with continue/abort; cherry-pick and revert also support root commits and merge commits with `--mainline`. |
| Stash | supported/limited | Tracked work plus optional untracked/ignored files, pathspec-limited push/create/apply/pop, store with optional message, pathspec-filtered read-only show/patch inspection, branch-from-stash, and clear. |
| Sparse checkout | supported/limited | Practical pattern-based sparse; sparse-index `sdir` directory placeholders are expanded on read, and mutating commands write a normal full v2 index rather than preserving sparse-index shape. |
| Shallow clone/fetch | supported/limited | Depth-based smart transport scope; partial-clone behavior is covered separately. |
| Hooks | supported/limited | `pre-commit`, `commit-msg`, `pre-merge-commit`, `post-commit`, `post-merge`, `post-checkout`, `pre-push`. |
| LFS | supported/limited | Merge worktree writes smudge valid pointer blobs to local `.git/lfs/objects` media when present, and can fetch missing media from local/file LFS stores, HTTP(S) Git LFS batch download endpoints, or SSH `git-lfs-authenticate` followed by the advertised HTTP(S) batch endpoint before materializing the merge result. Staging paths marked `filter=lfs` in root `.gitattributes` or `.git/info/attributes` stores media in `.git/lfs/objects` and writes canonical pointer blobs; restore/checkout materializes pointer blobs to media for `filter=lfs` paths when media is cached or fetchable through the same LFS transfer paths. Paths without `filter=lfs` still treat LFS pointers as ordinary files/blobs. |
| Partial clone | supported/limited | Repository format version 1 with `extensions.partialClone` is accepted. `clone --filter=SPEC` creates a partial clone: over HTTP/SSH the arbitrary `SPEC` is sent verbatim when the server advertises the upload-pack `filter` capability; over local transport version evaluates `blob:none` and `blob:limit=<n>` directly (selective object copy, richer specs are rejected for local sources). The clone is configured as a partial clone (`extensions.partialClone`, promisor remote, `partialclonefilter`). Missing objects are then lazily fetched once from the promisor by object id over local (single-object copy), HTTP, or SSH transports without updating refs. Promisor pack sidecars without a configured partial-clone remote still report the promisor limitation. |
| worktreeConfig | supported | `extensions.worktreeConfig` is accepted; the per-worktree `config.worktree` is layered over the common `config` when reading configuration (per-worktree values override the common config, and absent keys fall back to it). Writing to `config.worktree` (`config --worktree`) is not yet provided. |

Supported workflows include ordinary init/stage/save/status/log, Git inspection via `git fsck --strict`, local/file remotes, configured HTTP(S)/SSH remotes, practical branch/tag/replay/stash workflows, sparse checkouts, linked worktrees, and conservative submodule updates.

Partially supported workflows include reflog parity, worktree administrative parity beyond the documented per-worktree state, submodule URL rewriting and advanced recursive modes beyond local gitlink checkout/fetch/nested update, shallow deepening variants, advanced external merge driver edge cases, exact Git ort output/diagnostic parity beyond the documented success and conflict-summary diagnostics, conditional config discovery forms outside the documented ignore-source includeIf set, and advanced pathspec magic beyond the documented attr/set/unset/unspecified/value forms.

Interactive rebase (`rebase -i`) supports pick/drop/reorder/squash/fixup via the sequence editor; reword/edit/exec are not yet supported. Root rebase is supported via `rebase --root --onto NEWBASE` (bare `--root` is not). Merge-preserving rebase is supported via `rebase --rebase-merges` for two-parent merges (octopus merges are rejected; `--preserve-merges` is not supported). Unsupported workflows include server-side hook management and a website.


## Archive export

Supported/limited: `version archive` can export committed trees as TAR or ZIP, preserving file bytes, CRLF data, binary payloads, directory structure, symlink metadata, and executable metadata where the archive format supports it. Gitlink/submodule entries are represented as placeholder files and are not recursively archived; Git symlink entries are represented as symlinks in TAR/ZIP metadata. Byte-for-byte parity with `git archive` is not required. Supported archive prefixes are safe relative root paths applied to emitted entry names after pathspec matching. Unsupported: compressed archive formats, unsafe archive prefixes, export-subst attributes, Git LFS materialization, and remote archive streaming.

## Unsupported diagnostics

SHA-256 object-format repositories (`extensions.objectFormat = sha256`) are supported: `init --object-format=sha256` creates them, and they are read and written (local and smart HTTP/SSH transport, format negotiated via the `object-format=sha256` capability), byte-compatible with system git. A repository is a single object format for its lifetime.

Reftable ref storage (`extensions.refStorage = reftable`, `repositoryformatversion = 1`) is supported for reading and writing: version reads git-created reftable stacks, writes git-readable tables, stores the reflog in reftable log blocks, and `init --ref-format=reftable` creates such repositories (verified `fsck`-clean and `git reflog`-readable against git 2.54). Writes append one small table per transaction with geometric auto-compaction keeping the stack bounded, preserving tombstones and reflog history through merges.

Unsupported features must fail with precise, stable diagnostics. The cases include promisor sidecars without a configured partial-clone remote, unavailable HTTP/3 support, h2c upgrade, server push, unsupported remote URL schemes, and SSH upload-pack/receive-pack streaming limitations.
